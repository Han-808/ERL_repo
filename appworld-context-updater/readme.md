# Context Update Harness: Experiment Design

## Goal

Compare context update methods on the 90 AppWorld train tasks. Test whether proactive hypothesis-driven learning outperforms passive trace summarization, and how ACE ablations affect performance.

## Shared Infrastructure

### Data
- **Tasks**: `train` split (90 tasks), loaded via `appworld.task.load_task_ids("train")`
- **Traces**: Generated by running the agent per task. Each trace stores code + execution output per turn, and the final test report is stored separately in the record. Agent reasoning/planning text and `<think>` blocks are stripped before passing to context update models.
- **Ground truth**: Available via `task.ground_truth`. `compiled_solution_code` is stored in trace metadata. Only ACE variants with `use_ground_truth=True` receive it during context update.

### Agent
- Served via sglang (auto-launched unless `--skip-server-launch` / `--sglang-base-url` is given)
- Prompt: `libs/ace-appworld/experiments/prompts/appworld_react_code_agent_playbook_generator_prompt.txt` with `{{ playbook }}` slot filled by the current context
- Execution via `PlaybookReActAgent` (ReAct loop: write Python, execute, observe output, repeat up to `--max-steps`)

### Shared Input to Context Update

All methods receive the same task-side information:
- **Current context**: the full context text before this task
- **Task instruction**: the task the agent was asked to solve
- **Full trace**: code + execution results per turn (reasoning/`<think>` blocks stripped)
- **Test report**: the unit test result after the agent finished (pass/fail + details)
- **Success flag**: boolean
- **Ground truth code**: only passed to ACE variants with `use_ground_truth=True`

### Shared Loop (per method)
```
for task_i in train_tasks:
    1. Run agent on task_i with current context -> trace_i, test_report_i, result_i
    2. Update context using method-specific pipeline -> context_delta, new_context
    3. Append record to traces.jsonl
```

### Output Structure
```
outputs/<method_name>/
  traces.jsonl              # one JSON line per task
  latest_context.txt        # current context after last completed task
  context_step{N}.txt       # snapshots every --context-save-every tasks
  api_calls_agent.jsonl     # agent LLM call log
  api_calls_context.jsonl   # context-update LLM call log
```

Each `traces.jsonl` line:
```json
{
  "task_id": "...",
  "context_before": "full context text before this task",
  "full_trace": [
    {"turn": 1, "code": "...", "execution_result": "..."},
    {"turn": 2, "code": "...", "execution_result": "..."}
  ],
  "test_report": "...",
  "context_after": "full context text after update",
  "context_delta": "method-specific delta (e.g., summary diff, curator ops, new hypotheses)",
  "metadata": {
    "task_instruction": "...",
    "ground_truth_code": "...",
    "success": true,
    "num_steps": 10,
    "score": 0.0
  }
}
```

## Methods

### Method 1a: Summary Rewrite (`summary_v1`)

**Idea**: Single LLM call to compress previous context + new trace into a full replacement context.

**Pipeline**: 1 model call per task.

```
Input:  current_context + task_instruction + full_trace + test_report + success
Model:  Summarizer (free-form rewrite)
Output: updated_context (free-form text, replaces context verbatim)
```

**Context format**: Free-form text. No structural constraints.

**context_delta**: the full new context (since it's a complete rewrite).

---

### Method 1b: Summary Delta (`summary_delta_v1`)

**Idea**: Single LLM call that reads the current context + trace and outputs structured delta operations (edit/add/delete) on tagged items.

**Pipeline**: 1 model call per task.

```
Input:  current_context + task_instruction + full_trace + test_report + success
Model:  Summarizer-Delta (outputs JSON operations)
Output: list of delta operations applied to tagged items
```

**Context format**: Flat list of tagged items:
```
[item-00001] Always paginate through all pages using while True loop.
[item-00002] Use supervisor app to get account credentials.
```

**Delta operations**: `edit` (revise existing item), `add` (insert new item, optional `after_tag` for position), `delete` (remove item).

**context_delta**: the normalized operations list.

---

### Method 2: ACE (`ace` and ablation variants)

**Idea**: Reproduce the ACE pipeline (Reflector + Curator). The Reflector diagnoses the trajectory; the Curator adds structured bullets to a sectioned playbook.

**Pipeline**: 2 model calls per task.

```
Call 1: Reflector  ->  reflection JSON (reasoning, error_id, root_cause, correct_approach, key_insight)
Call 2: Curator    ->  ADD operations on playbook sections
Output: updated sectioned playbook
```

**Context format**: Sectioned playbook with 8 standard sections (`strategies_and_hard_rules`, `apis_to_use_for_specific_information`, `useful_code_snippets_and_templates`, `common_mistakes_and_correct_strategies`, `problem_solving_heuristics_and_workflows`, `verification_checklist`, `troubleshooting_and_pitfalls`, `others`). Bullets have section-specific ID prefixes (e.g., `shr-00001`, `api-00002`).

**Ablation variants** (configured via `ACEModel` constructor):

| run.py key        | Ground truth | Test report | Initial playbook |
|-------------------|:---:|:---:|---|
| `ace`             | yes | yes | default (initial) |
| `ace_nogt`        | no  | yes | default |
| `ace_notest`      | yes | no  | default |
| `ace_pb_empty`    | yes | yes | empty |
| `ace_pb_null`     | yes | yes | null |
| `ace_nogt_notest` | no  | no  | default |

**context_delta**: the curator operations list.

---

### Method 2b: ACE-AED (`ace_aed`)

**Idea**: Same reflector as ACE, but the curator supports ADD, EDIT, and DELETE operations (not just ADD). Starts from the empty playbook and does not use ground truth.

**Pipeline**: 2 model calls per task.

```
Call 1: Reflector   ->  reflection JSON (same as ACE, no GT)
Call 2: AED Curator ->  ADD/EDIT/DELETE operations on playbook sections
Output: updated sectioned playbook
```

The AED curator can refine existing bullets with new evidence (EDIT) and remove outdated or incorrect bullets (DELETE), unlike the ADD-only ACE curator.

**context_delta**: the curator operations list.

---

### Method 2c: Hypothesis V3 (`hypothesis_v3`, `hypothesis_v3_gt`)

**Idea**: Use the exact ACE pipeline for the 8 standard playbook sections, then add a third dedicated LLM call — the Open Question Curator — that manages a separate `## OPEN QUESTIONS` section. Unlike the ACE curator (ADD-only), the OQ Curator can ADD new questions, EDIT existing ones when evidence sharpens or redirects them, and DELETE questions that have been fully resolved.

**Key difference from Hypothesis V1/V2**: V1/V2 replace the entire knowledge structure with an observation+question notebook. V3 keeps the ACE playbook unchanged for declarative knowledge, and adds open questions as a lightweight forward-looking layer on top. The two concerns stay cleanly separated.

**Key difference from ACE-AED**: ACE-AED's curator edits/deletes existing playbook bullets. V3 keeps the ACE curator ADD-only for the 8 standard sections, and introduces a separate focused curator that exclusively manages open questions.

**Pipeline**: 3 model calls per task.

```
Call 1: Reflector    ->  reflection JSON (identical to ACE)
Call 2: ACE Curator  ->  ADD operations on the 8 standard playbook sections (identical to ACE)
Call 3: OQ Curator   ->  ADD/EDIT/DELETE on ## OPEN QUESTIONS section only
Output: playbook from Call 2  +  ## OPEN QUESTIONS section from Call 3
```

**Context format**: Standard ACE sectioned playbook with `## OPEN QUESTIONS` appended. Open question bullets use `oq-NNNNN` IDs:

```
## STRATEGIES AND HARD RULES
[shr-00001] ...

## APIS TO USE FOR SPECIFIC INFORMATION
[api-00001] ...

...

## OPEN QUESTIONS
[oq-00001] Does search_contacts need `relationship` or does `query` alone suffice? ...
[oq-00002] Does like_transaction raise or silently skip already-liked items? ...
```

**Call 3 — OQ Curator** receives 5 inputs:
1. Task instruction
2. Current full playbook (main sections + current OQ section)
3. Current generated attempt (conversation history)
4. Reflection JSON from Call 1
5. Playbook delta — the ADD operations just applied to the main sections by Call 2

It focuses exclusively on the OQ section with ADD/EDIT/DELETE. The playbook delta lets the OQ curator see what new knowledge was just added, so it avoids adding redundant open questions for things now explicitly documented.

Good open questions are forward-facing: they describe what to try or verify next time, not just what is unknown.

**Variants**: `hypothesis_v3` uses ground truth (default); `hypothesis_v3_nogt` does not.

**context_delta**: `{"ace_curator": [...], "oq_curator": [...]}` — operations from both curators.

### Method 3a: Hypothesis V1 (`hypothesis_v1`)

**Core idea**: Instead of declarative knowledge (facts from past traces), maintain a notebook with two sections: **observations** (what is known, with evidence) and **open questions** (what to investigate next). This guides the agent to actively learn about the environment.

**Pipeline**: 3 model calls per task.

```
Call 1: Observation Extractor  ->  observations grounded in trace evidence
Call 2: Question Generator     ->  open questions about the environment
Call 3: Notebook Integrator    ->  add/edit/delete operations on the notebook
Output: updated notebook (## OBSERVATIONS + ## OPEN QUESTIONS)
```

**Context format**: Tagged items organized into two sections:
```
## OBSERVATIONS
[observation-00001] Always paginate before filtering: Spotify's show_liked_songs returns one page per call...

## OPEN QUESTIONS
[question-00001] Check if search_contacts needs relationship param: ...
```

**Key difference from ACE**: ACE produces declarative rules ("use Phone contacts for relationships"). The notebook records observations with evidence and uncertainty, plus open questions that guide future exploration.

**context_delta**: the normalized operations list.

---

### Method 3b: Hypothesis V2 (`hypothesis_v2`)

Same 3-call pipeline as V1 but with improved prompts:
- Observation and question prompts produce natural-language analytical notes (not notebook items), which are then rewritten by the integrator
- Prompts include worked examples showing what good observations/questions look like
- Notebook items use shorter prefixes (`obs-*`, `que-*`) and custom parsing/rendering logic

**context_delta**: the normalized operations list.

## CLI Reference

```bash
python run.py \
  --method <method_name> \
  --model-name <hf_model_path> \
  [--context-model-name <model>]     # separate model for context updates (default: same as agent)
  [--max-steps 40]                   # max ReAct turns per task
  [--temperature 0.6]                # sampling temperature
  [--max-tokens 8192]                # max generation tokens
  [--dataset train]                  # task split
  [--task-ids ID1 ID2 ...]           # run specific tasks
  [--task-limit N]                   # limit number of tasks
  [--task-offset N]                  # skip first N tasks
  [--task-seed 100]                  # shuffle seed for task order
  [--experiment-name NAME]           # appworld experiment name (default: method name)
  [--output-dir DIR]                 # method outputs directory
  [--experiment-outputs-dir ./outputs]  # appworld experiment outputs base
  [--context-save-every 10]          # snapshot context every N tasks
  [--sglang-base-url URL]            # skip server launch, use existing endpoint
  [--skip-server-launch]             # don't auto-launch sglang
  [--sglang-host 127.0.0.1]         # sglang bind host
  [--tp 1]                           # tensor parallelism
  [--dp 1]                           # data parallelism
  [--sglang-extra-args ...]          # extra args passed to sglang
  [--random-seed 100]                # appworld random seed
```

Available methods: `summary_v1`, `summary_delta_v1`, `ace`, `ace_nogt`, `ace_notest`, `ace_pb_empty`, `ace_pb_null`, `ace_nogt_notest`, `ace_aed`, `hypothesis_v1`, `hypothesis_v2`, `hypothesis_v3`, `hypothesis_v3_nogt`

### Example: run.sh

`run.sh` runs a batch of methods indexed by `$ID` across multiple model sizes:

```bash
ID=0 bash run.sh   # runs ace with Qwen3-4B, 8B, 14B, 32B sequentially
ID=1 bash run.sh   # runs ace_nogt
ID=2 bash run.sh   # runs ace_notest
ID=3 bash run.sh   # runs ace_pb_empty
```

## Evaluation

After all 90 train tasks, compare methods on:

1. **Online accuracy curve**: pass rate as a function of task index (learning speed)
2. **Final context quality**: freeze each method's final context, evaluate on `test_normal` with `scripts/appworld-eval/eval_static_playbook.py`
3. **Context efficiency**: context length (tokens) vs. accuracy tradeoff
4. **Qualitative**: inspect contexts for actionable vs. generic content

## Key Differences Between Methods

| Aspect | Summary Rewrite | Summary Delta | ACE | ACE-AED | Hypothesis V3 | Hypothesis V1/V2 |
|--------|:-:|:-:|:-:|:-:|:-:|:-:|
| Context structure | Free-form | Tagged items | Sectioned playbook | Sectioned playbook | Sectioned playbook + OQ section | Tagged items (obs + questions) |
| Main update mechanism | Full rewrite | edit/add/delete | ADD only | ADD/EDIT/DELETE | ADD only (+ OQ ADD/EDIT/DELETE) | edit/add/delete |
| Knowledge type | Declarative | Declarative | Declarative | Declarative | Declarative + open questions | Observations + open questions |
| Uses ground truth | No | No | Configurable | No | Configurable | No |
| Forward-looking | No | No | No | No | Yes (OQ section) | Yes (what to investigate next) |
| Calls per task | 1 | 1 | 2 | 2 | 3 | 3 |
